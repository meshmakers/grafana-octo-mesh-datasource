# Deployment Guide: Tenant-Specific OAuth Backend

## Prerequisites

### 1. Identity Server Client Registration

Register a `grafana-datasource` client in each OctoMesh tenant where Grafana users need access.

**Client Configuration:**

| Setting | Value |
|---------|-------|
| Client ID | `grafana-datasource` |
| Grant Type | `authorization_code` |
| PKCE | Required (S256) |
| Client Secret | None (public client) |
| Allow Offline Access | `true` |
| Scopes | `openid`, `profile`, `email`, `octo_api`, `offline_access` |
| Redirect URIs | `https://grafana.{env}.mm.cloud/api/datasources/proxy/uid/{datasourceUID}/auth/callback` |

**Note on Redirect URI:** The datasource UID is generated by Grafana when the datasource is created. After creating the datasource, find its UID in Grafana (Settings > Data Sources > OctoMesh > URL shows `/datasources/edit/{uid}`) and register the callback URL.

Alternatively, register a wildcard callback pattern if the Identity Server supports it:
```
https://grafana.{env}.mm.cloud/api/datasources/proxy/uid/*/auth/callback
```

### 2. Grafana Helm Values Update

Update the OctoMesh datasource provisioning in `grafana-values-base.yaml`:

**Before (oauthPassThru):**
```yaml
- name: OctoMesh
  type: grafana-octo-mesh-datasource
  url: http://octo-mesh-asset-rep-services.octo.svc.cluster.local
  access: proxy
  jsonData:
    oauthPassThru: true
```

**After (backend OAuth):**
```yaml
- name: OctoMesh
  type: grafana-octo-mesh-datasource
  url: http://octo-mesh-asset-rep-services.octo.svc.cluster.local
  access: proxy
  jsonData:
    tenantId: 'meshtest'
    identityServerUrl: 'https://connect.{env}.mm.cloud'
    oauthClientId: 'grafana-datasource'
    oauthScopes: 'openid profile email octo_api offline_access'
```

**Per-environment Identity Server URLs:**

| Environment | Identity Server URL |
|-------------|-------------------|
| test-2 | `https://connect.test-2.mm.cloud` |
| staging | `https://connect.staging.mm.cloud` |
| production | `https://connect.production.mm.cloud` |

### 3. Multiple Tenants per Grafana Instance

For multiple tenants, create separate Grafana Organizations with dedicated datasource instances:

```yaml
# Org 1: meshtest
- name: OctoMesh (meshtest)
  type: grafana-octo-mesh-datasource
  orgId: 1
  jsonData:
    tenantId: 'meshtest'
    identityServerUrl: 'https://connect.test-2.mm.cloud'
    oauthClientId: 'grafana-datasource'

# Org 2: sbeg
- name: OctoMesh (sbeg)
  type: grafana-octo-mesh-datasource
  orgId: 2
  jsonData:
    tenantId: 'sbeg'
    identityServerUrl: 'https://connect.test-2.mm.cloud'
    oauthClientId: 'grafana-datasource'
```

## Authentication Flow

1. User opens a dashboard with an OctoMesh datasource
2. QueryEditor shows "Authentication required" banner
3. User clicks "Authenticate" — popup opens to Identity Server login
4. Identity Server receives `acr_values=tenant:{tenantId}` and redirects to tenant login
5. After login, callback returns to plugin backend with authorization code
6. Backend exchanges code for tokens (PKCE), caches per user/tenant
7. Subsequent requests use cached token (auto-refresh via refresh token)
8. When switching Grafana Org (= different tenant), user authenticates once for that tenant
9. After initial login per tenant, SSO cookies enable silent re-authentication

## Token Lifecycle

- **Access Token**: 3600s default, 30s pre-expiry buffer
- **Refresh Token**: Used automatically when access token expires
- **Cache**: In-memory per Grafana pod. Lost on pod restart — users re-authenticate via SSO (no password needed if IdP session cookie is valid)

## Rollback

To revert to the old `oauthPassThru` approach:
1. Restore `oauthPassThru: true` in datasource jsonData
2. Remove `backend: true` and `executable` from plugin.json
3. Restore `routes` in plugin.json
4. Deploy the previous plugin version (without Go binary)
